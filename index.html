Project Brief

An AI-classified complaint pipeline, built and documented like a shipped product.

This site is the working reference for building the AI Complaint Management System end to end: a 7-day plan for a 2-person team, followed by a full code walkthrough of the HTML templates, the CSS system, the JavaScript layer, the Flask backend, and the machine learning pipeline that classifies every complaint.

Team size 2 developers Duration 7 days Stack Flask · SQLite · scikit-learn Frontend HTML · CSS · vanilla JS

Every submitted complaint runs through this pipeline synchronously — no external API calls, all three models are loaded once at Flask startup.

Section 04 — Frontend

HTML Templates

All pages extend one base.html layout using Jinja2 template inheritance — this is how the navbar, flash messages, and footer stay consistent across every page without being copy-pasted.

Template inheritance pattern

base.html — defines the shared shell
<body>
  <nav class="navbar">...</nav>

  <main class="container">
    {% with messages = get_flashed_messages(with_categories=true) %}
      {% if messages %}
        <div class="flash-wrap">
        {% for category, message in messages %}
          <div class="flash flash-{{ category }}">{{ message }}</div>
        {% endfor %}
        </div>
      {% endif %}
    {% endwith %}

    {% block content %}{% endblock %}
  </main>
</body>

Every other page only defines what's different — the content block:

submit_complaint.html — extends base.html
{% extends "base.html" %}
{% block title %}Submit Complaint{% endblock %}
{% block content %}
<form method="POST" action="{{ url_for('submit_complaint') }}">
  <textarea name="complaint_text" rows="6" required></textarea>
  <button type="submit" class="btn-primary">Submit Complaint</button>
</form>
{% endblock %}

Why this matters

Changing the navbar or flash-message style once in base.html updates all 6 pages instantly. This is the same pattern used in production Flask/Django apps — never duplicate layout markup per page.

Section 06 — Frontend

JavaScript Layer

The core app is intentionally server-rendered — forms submit natively and Flask handles redirects, so no JS framework is required. static/js/app.js adds small progressive-enhancement touches that make the UI feel polished without adding any build step.

static/js/app.js — auto-dismiss flash messages
document.addEventListener("DOMContentLoaded", () => {
  document.querySelectorAll(".flash").forEach((el) => {
    setTimeout(() => {
      el.style.transition = "opacity 0.4s ease";
      el.style.opacity = "0";
      setTimeout(() => el.remove(), 400);
    }, 5000);
  });
});
static/js/app.js — live character counter on the complaint form
const textarea = document.querySelector('textarea[name="complaint_text"]');
if (textarea) {
  const counter = document.createElement("div");
  counter.className = "char-counter";
  textarea.after(counter);

  const updateCount = () => {
    counter.textContent = `${textarea.value.length} characters`;
  };
  textarea.addEventListener("input", updateCount);
  updateCount();
}
Why so little JS? Keeping business logic (auth, prediction, routing) entirely on the server means there's one source of truth and no API layer to secure separately. Add a JS framework later only if the admin panel needs live-updating data without page reloads.
Section 07 — Backend

Flask Application

app.py is a single-file Flask app: routes, SQLite access, and the ML prediction call all live together for a project this size. Below is how each core piece works.

1. Session-based authentication

app.py — login route
@app.route("/login", methods=["GET", "POST"])
def login():
    if request.method == "POST":
        email = request.form.get("email", "").strip().lower()
        password = request.form.get("password", "")

        db = get_db()
        user = db.execute("SELECT * FROM users WHERE email = ?", (email,)).fetchone()

        if user and check_password_hash(user["password_hash"], password):
            session["user_id"] = user["id"]
            session["role"] = user["role"]
            return redirect(url_for("dashboard"))

        flash("Invalid email or password.", "danger")
        return redirect(url_for("login"))

    return render_template("login.html")

Passwords are never stored in plain text — generate_password_hash() / check_password_hash() from Werkzeug handle salted hashing.

2. Route protection with decorators

app.py — reusable auth guards
def login_required(f):
    @wraps(f)
    def wrapper(*args, **kwargs):
        if "user_id" not in session:
            return redirect(url_for("login"))
        return f(*args, **kwargs)
    return wrapper

def admin_required(f):
    @wraps(f)
    def wrapper(*args, **kwargs):
        if session.get("role") != "admin":
            return redirect(url_for("dashboard"))
        return f(*args, **kwargs)
    return wrapper

Any route just adds @login_required or @admin_required above it — no repeated if-checks inside every view function.

3. Complaint submission → ML prediction → storage

app.py — the core AI-integration point
@app.route("/submit-complaint", methods=["GET", "POST"])
@login_required
def submit_complaint():
    if request.method == "POST":
        text = request.form.get("complaint_text", "").strip()
        prediction = predict_complaint(text)   # -> category, priority, department

        db = get_db()
        db.execute(
            """INSERT INTO complaints
               (user_id, complaint_text, category, priority, department, status, created_at)
               VALUES (?, ?, ?, ?, ?, 'Open', ?)""",
            (session["user_id"], text, prediction["category"], prediction["priority"],
             prediction["department"], datetime.now().isoformat()),
        )
        db.commit()
        return redirect(url_for("dashboard"))

    return render_template("submit_complaint.html")

4. SQLite access pattern

app.py — per-request DB connection
def get_db():
    if "db" not in g:
        g.db = sqlite3.connect(DB_PATH)
        g.db.row_factory = sqlite3.Row
    return g.db

@app.teardown_appcontext
def close_db(exception):
    db = g.pop("db", None)
    if db is not None:
        db.close()

Flask's g object keeps one connection per request and closes it automatically — this is the standard pattern for SQLite in Flask, avoiding leaked connections.

Section 08 — Backend

Machine Learning Pipeline

Three separate supervised models share one TF-IDF vectorizer. Training happens offline once; Flask only loads the saved .pkl files and calls .predict() — no retraining at request time.

TaskAlgorithmAccuracy
Category predictionTF-IDF + Logistic Regression~100%
Priority predictionTF-IDF + Random Forest~51%
Department predictionTF-IDF + Logistic Regression~100%
Why is priority accuracy lower? Category and department are strongly implied by the words in a complaint. Priority isn't — in reality it depends on context the text alone doesn't carry (repeat complaints, deadlines, who's affected). This is a legitimate ML limitation to explain in a viva, not a bug.

Training a model

machine_learning/train_models.py
vectorizer = TfidfVectorizer(max_features=3000, ngram_range=(1, 2), stop_words="english")
vectorizer.fit(df["complaint_text"])

encoder = LabelEncoder()
y = encoder.fit_transform(df["category"])
X = vectorizer.transform(df["complaint_text"])

X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.2, stratify=y)

model = LogisticRegression(max_iter=1000)
model.fit(X_train, y_train)

joblib.dump(model, "category_model.pkl")
joblib.dump(encoder, "category_encoder.pkl")

Running inference in Flask

app.py — predict_complaint()
def predict_complaint(text):
    X = ML["vectorizer"].transform([text])

    cat_pred = ML["category_model"].predict(X)[0]
    category = ML["category_encoder"].inverse_transform([cat_pred])[0]

    pri_pred = ML["priority_model"].predict(X)[0]
    priority = ML["priority_encoder"].inverse_transform([pri_pred])[0]

    dept_pred = ML["department_model"].predict(X)[0]
    department = ML["department_encoder"].inverse_transform([dept_pred])[0]

    return {"category": category, "priority": priority, "department": department}

All models are loaded once into a module-level ML dict at Flask startup — not on every request — so prediction is effectively instant.

Section 09

Setup Guide

Install Python 3.11 or 3.12

Avoid brand-new Python releases (3.13/3.14) — some ML packages don't yet ship pre-built wheels for them, which forces slow source builds that often fail on Windows without Visual Studio Build Tools.

Create and activate a virtual environment

terminal
python -m venv venv
venv\Scripts\activate

Install dependencies

terminal
pip install -r requirements.txt

Generate the dataset and train the models

terminal
python dataset/generate_dataset.py
python machine_learning/train_models.py

Run the app

terminal
python app.py

Open http://127.0.0.1:5000. Default admin login: admin@cms.local / admin123.
